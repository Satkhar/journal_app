# План развития карточек участников

## Зафиксированная основа

- `Participant` — глобальная сущность со стабильным UUID.
- `month_participants` — участие человека в конкретном месяце.
- `attendance` ссылается на membership через составной FK.
- Удаление из месяца не удаляет глобального участника.
- Local schema version — `PRAGMA user_version = 11`.
- Versioned local schema v2-v10 мигрирует последовательно до v11; remote — до
  v9.
- Промежуточная development-v6 с `is_trainer` чинится при открытии; колонка
  удаляется, прежние значения сохраняются в compatibility-таблице.
- Unversioned remote legacy schema автоматически не мигрируется.

## Этап 2. Профиль участника — выполнен

### Зафиксированный контракт даты рождения

- дата рождения целиком может быть неизвестна;
- если день рождения указан, день и месяц обязательны вместе;
- год nullable;
- в БД: nullable `birth_day`, `birth_month`, `birth_year`;
- полная календарная валидность проверяется domain/use-case слоем до SQL.

Историчное имя необязательно; при его отсутствии таблица показывает ФИО.
Редактируемый профиль требует хотя бы одно из этих двух полей. Один контакт
необязателен. Контакт — opaque строка для VK, Telegram или
телефона, без автоматического открытия. Заметка — plain text, максимум 4096
символов. Кто вёл тренировку — свойство участника в конкретную дату; оно
хранится как `LedTraining` в `participant_day_markers`, а не в профиле.

### Реализовано

1. Schema migration v2 -> v3 и fresh schema v3.
2. Добавить типизированные поля профиля:
   - дата рождения с точностью;
   - ФИО;
   - контакт;
   - заметка;
   - `archived_at` уже заложен в v2.
3. Добавить storage API чтения/редактирования профиля по `ParticipantId`.
4. Добавить use cases:
   - открыть карточку;
   - изменить профиль;
   - архивировать/восстановить.
5. Добавить `ParticipantDialog`.
6. Открывать карточку двойным кликом по строке.
7. Запретить физическое удаление участника из обычного UI.

### Проверка

- Одинаковые отображаемые имена не конфликтуют.
- Переименование не меняет UUID и историю посещений.
- Архивирование не удаляет membership и attendance прошлых месяцев.
- Невалидная дата рождения отклоняется до SQL.

## Этап 3. Локальный герб — выполнен

### Текущий контракт

1. Schema migration v10 -> v11 добавляет `participant_emblems`.
2. У участника может быть один герб. Запись связана с глобальным
   `ParticipantId` и удаляется каскадно вместе с участником.
3. Исходное изображение проверяется декодером, уменьшается максимум до
   1024x1024 и сохраняется как нормализованный PNG BLOB размером до 5 MiB.
   Пользовательский путь и EXIF в БД не сохраняются.
4. Таблица хранит PNG, SHA-256, исходное имя файла, размеры и `revision`.
5. Герб загружается отдельно от списка и профиля: большие бинарные данные не
   входят в обычный read-model карточки.
6. Замена или удаление герба использует compare-and-swap по `revision`.
7. Профиль и выбранное действие с гербом сохраняются одной SQLite-транзакцией.
   Ошибка любой части откатывает всю карточку.

### Ограничения

- Герб доступен только в local SQLite. `MonthSnapshot`, `JournalRemote` и
  текущий sync-контракт его не передают.
- Это заготовка для одного герба, не фотогалерея. Multi-photo, thumbnails,
  отдельный asset-каталог и бинарный asset transport остаются будущей работой.
- Backup локальной SQLite уже включает текущий PNG BLOB. При выносе файлов в
  asset-каталог backup/restore должен охватывать БД и assets вместе.

## Этап 3.1. Замеры ударов на время — выполнен

- Schema v11 добавляет `participant_strike_tests` в основную локальную БД.
- Каждый замер хранит собственный UUID, `ParticipantId`, время, руку,
  количество ударов, длительность в секундах, оружие (`Меч` или `Тямбара`),
  примечание и `revision`.
- Сохраняются исходные факты замера. Скорость ударов в секунду и минуту
  вычисляется при чтении и не дублируется в БД.
- История редактируется и удаляется с compare-and-swap по `revision`.
- Прогресс сравнивается только для одинаковых руки, оружия и длительности.
  Сравнивать, например, 30 секунд мечом и 60 секунд тямбарой как одну серию
  нельзя.
- Замеры доступны только локально и пока не входят в remote sync.

## Этап 4. Синхронизация профилей, замеров и assets

1. Добавить protocol/schema version handshake.
2. Спроектировать server-side `revision`, `updated_at` и tombstone для
   участников, гербов и замеров ударов. Локальный CAS этих записей не является
   готовым sync-контрактом.
3. Синхронизировать UUID и метаданные отдельно от month snapshot.
4. Добавить отдельный бинарный asset transport; не передавать гербы или фото
   как SQL text.
5. Проверять SHA-256 после передачи.
6. Перевести remote API с вложенного `QEventLoop` на async.
7. Добавить conflict detection и явную политику merge.
8. До передачи персональных данных обязательны auth и TLS.
9. Тестировать interrupted upload, retry, duplicate delivery, conflict и deletion.

## Этап 5. Расширяемые поля

Не внедрять EAV заранее.

Порядок:

1. Собрать реальные требования после эксплуатации типизированного профиля.
2. Если набор полей действительно меняется между установками, добавить:
   - definitions;
   - явные типы;
   - validation rules;
   - typed values;
   - schema/version rules для sync.
3. Не хранить все значения одной невалидируемой строкой.

## Этап 6. Продуктовые функции

1. Поиск и фильтры.
2. Напоминания о днях рождения.
3. Теги/категории.
4. История и статистика посещений.
5. Поиск возможных дублей без автоматического слияния.
6. Export/import.
7. Backup/restore с проверкой БД и будущих внешних assets.
8. Audit log для изменений профиля.

## Порядок исполнения

Следующий порядок: `Этап 4 -> Этап 5 -> Этап 6`. Локальный этап 3 выполнен;
multi-photo при подтверждённой необходимости станет отдельным расширением
asset-модели.

Этап 4 нельзя считать production-ready без auth/TLS. Этап 5 нельзя начинать
без подтвержденного набора пользовательских custom fields.
