# План развития карточек участников

## Зафиксированная основа

- `Participant` — глобальная сущность со стабильным UUID.
- `month_participants` — участие человека в конкретном месяце.
- `attendance` ссылается на membership через составной FK.
- Удаление из месяца не удаляет глобального участника.
- Local schema version — `PRAGMA user_version = 2`.
- Старые development-БД не мигрируются: приложение отклоняет их явно.
- Remote legacy schema автоматически не мигрируется.

## Этап 2. Профиль участника

### Зафиксированный контракт даты рождения

- дата рождения целиком может быть неизвестна;
- если день рождения указан, день и месяц обязательны вместе;
- год nullable;
- в БД: nullable `birth_day`, `birth_month`, `birth_year`;
- полная календарная валидность проверяется domain/use-case слоем до SQL.

До реализации остается определить только набор контактов и формат заметок.
### Реализация

1. Schema migration v2 -> v3.
2. Добавить типизированные поля профиля:
   - дата рождения с точностью;
   - заметка;
   - `archived_at` уже заложен в v2.
3. Добавить storage API чтения/редактирования профиля по `ParticipantId`.
4. Добавить use cases:
   - открыть карточку;
   - изменить профиль;
   - архивировать/восстановить.
5. Добавить `ParticipantDialog`.
6. Открывать карточку двойным кликом по строке.
7. Запретить физическое удаление участника из обычного UI.

### Проверка

- Одинаковые отображаемые имена не конфликтуют.
- Переименование не меняет UUID и историю посещений.
- Архивирование не удаляет membership и attendance прошлых месяцев.
- Невалидная дата рождения отклоняется до SQL.

## Этап 3. Фотографии

### Модель

1. Schema migration v3 -> v4: `participant_photos`.
2. В БД хранить метаданные:
   - photo UUID;
   - participant UUID;
   - storage key;
   - SHA-256;
   - MIME;
   - byte size;
   - width/height;
   - sort order;
   - primary flag.
3. Байты хранить в управляемом asset-каталоге, не по пользовательскому пути.

### Безопасность и надежность

1. Разрешить только заданные форматы.
2. Проверять формат декодером, не расширением файла.
3. Ограничить размер файла, разрешение и decoded memory.
4. Генерировать thumbnail; не декодировать оригинал при каждой отрисовке.
5. Запись: temporary file, validation, atomic rename, DB transaction.
6. Удаление: сначала пометить asset, затем garbage collection.
7. Проверить backup/restore БД вместе с asset-каталогом.

### UI

- Одна primary photo в таблице.
- Галерея в карточке.
- Добавление, выбор primary, изменение порядка, удаление.
- Lazy loading thumbnails.

## Этап 4. Синхронизация профилей и assets

1. Добавить protocol/schema version handshake.
2. Добавить `revision`, `updated_at`, tombstone для участников и фотографий.
3. Синхронизировать UUID и метаданные отдельно от month snapshot.
4. Добавить отдельный бинарный asset transport; не передавать фото как SQL text.
5. Проверять SHA-256 после передачи.
6. Перевести remote API с вложенного `QEventLoop` на async.
7. Добавить conflict detection и явную политику merge.
8. До передачи персональных данных обязательны auth и TLS.
9. Тестировать interrupted upload, retry, duplicate delivery, conflict и deletion.

## Этап 5. Расширяемые поля

Не внедрять EAV заранее.

Порядок:

1. Собрать реальные требования после эксплуатации типизированного профиля.
2. Если набор полей действительно меняется между установками, добавить:
   - definitions;
   - явные типы;
   - validation rules;
   - typed values;
   - schema/version rules для sync.
3. Не хранить все значения одной невалидируемой строкой.

## Этап 6. Продуктовые функции

1. Поиск и фильтры.
2. Напоминания о днях рождения.
3. Теги/категории.
4. История и статистика посещений.
5. Поиск возможных дублей без автоматического слияния.
6. Export/import.
7. Backup/restore с проверкой assets.
8. Audit log для изменений профиля.

## Порядок исполнения

`Этап 2 -> Этап 3 -> Этап 4 -> Этап 5 -> Этап 6`.

Этап 4 нельзя считать production-ready без auth/TLS. Этап 5 нельзя начинать
без подтвержденного набора пользовательских custom fields.
