# Architecture

## Слои

```mermaid
flowchart LR
  UI[MainWindow / Qt Widgets] --> APP[JournalApp]
  APP --> ST[IJournalStorage]
  ST --> LOCAL[JournalLocal]
  LOCAL --> SQLITE[SqliteConnect]
  SQLITE --> LDB[(SQLite schema v12)]
  UI --> SYNC[SyncService]
  SYNC --> SNAP[IMonthSnapshotStore]
  LOCAL -. implements .-> SNAP
  REMOTE[JournalRemote] -. implements .-> SNAP
  REMOTE --> API[libsql HTTP API]
  API --> RDB[(Remote schema v9)]
  UI -. diagnostic read-only .-> REMOTE
  UI --> EAPP[EventApp]
  EAPP --> EST[IEventStorage]
  EST --> ESQL[EventSqliteStorage]
  ESQL --> EDB[(events_data.db, schema v5)]
```

`MainWindow` отвечает за widgets и запуск use cases. Рабочий режим local-first:
`JournalApp` редактирует локальный `IJournalStorage`, а `SyncService` переносит
полный snapshot через узкий `IMonthSnapshotStore`. `SyncService` не знает URL,
HTTP или concrete adapters. SQL находится в `SqliteConnect`/`JournalRemote`.
Прямой remote-view оставлен только для диагностики PoC и не является вторым
source of truth.

## Стратегия жизненного цикла данных

Одна логическая база клуба хранит данные за все годы. Год и месяц являются
атрибутами записей, а не границами файлов: отдельные SQLite-файлы по годам и
ручной выбор «рабочей БД» запрещены архитектурным решением. Это сохраняет
единую карточку участника, сквозную историю, внешние ключи и транзакционные
границы.

Текущие `test_data.db` и `events_data.db` физически разделены, но это не
целевая модель будущей клубной системы. Bounded contexts и storage interfaces
должны остаться раздельными даже при последующем размещении их таблиц в одной
логической базе. Полное решение, причины и направление миграции описаны в
[стратегии хранения данных](data_storage_strategy.md).

## Доменная модель

### `Participant`

Глобальный человек:

- `ParticipantId` — стабильный UUID;
- `displayName` — вычисленное имя для журнала: историчное имя, если оно
  заполнено, иначе ФИО.

Имя не является ключом. Одинаковые имена допустимы.

### `ParticipantProfile`

Отдельная глобальная карточка: optional историчное имя, ФИО, один opaque contact
(VK, Telegram или телефон), nullable birthday, типизированное звание,
боевая рука, optional месяц начала тренировок, optional дата вступления в
клуб, набор пройденных званий с независимыми optional датами, plain-text notes
и `archived_at`. Birthday хранится как day/month и optional year; фиктивный
год не используется. Начало тренировок задаётся вручную с точностью до месяца,
чтобы не приписывать вымышленный день людям с историей до появления журнала.
Карточка не принимает будущий месяц. Когда статистика журнала доступна,
редактор явно предупреждает о месяце позже первого записанного посещения.
Это намеренно не SQL-инвариант: профиль и исторические month snapshots могут
импортироваться или синхронизироваться независимо, и описательное поле не
должно блокировать восстановление посещений. Отображаемый стаж приблизительный:
точность выше календарного месяца из этих данных получить невозможно.
Archive не удаляет membership или attendance. Начальный порядок званий: паж,
оруженосец, новичок, рекрут, гость, рыцарь. Звание остаётся отдельным полем и
визуальным признаком. Списки группируются по текущему званию и сортируются по
отображаемому имени внутри группы.

В основной таблице звания имеют отдельные, не чередующиеся цвета. Боевая рука
отображается компактной колонкой `ЛЕВ`/`ПРАВ`/`—`. Нижняя строка таблицы
содержит число посетителей каждого активного дня; среднее под таблицей делит
сумму этих значений на полное число активных дней, поэтому нулевые дни не
исчезают из знаменателя.

Текущее звание и история званий имеют разную семантику. Текущее поле `rank`
нужно для представления и группировки. `participant_rank_history` допускает
несколько строк на участника: отсутствие строки означает, что звание не
проходили; строка с `NULL obtained_on` означает полученное звание с неизвестной
датой. `guest` — статус посетителя, поэтому в историю получаемых званий не
записывается.

### `ParticipantEmblem`

Один optional локальный герб, связанный со стабильным `ParticipantId`. UI
проверяет изображение декодером, уменьшает максимум до 1024x1024 и сохраняет
нормализованный PNG BLOB до 5 MiB. Пользовательский абсолютный путь и EXIF не
являются данными модели. SHA-256 защищает целостность, `revision` обеспечивает
compare-and-swap при замене и удалении.

Герб загружается лениво отдельным storage-вызовом, поэтому каталог и обычное
чтение профиля не переносят BLOB. Изменение профиля и действие с гербом
фиксируются одной SQLite-транзакцией. Текущий remote adapter и `MonthSnapshot`
герб не поддерживают; multi-photo и отдельный asset transport остаются
будущей работой.

### `TimedStrikeTest`

Локальный замер ударов на время: отдельный UUID, `ParticipantId`, время в UTC,
рука, количество ударов, длительность в секундах, оружие (`Меч` или
`Тямбара`), plain-text примечание и `revision`. Скорость ударов в секунду и
минуту вычисляется из количества и длительности, а не хранится отдельным
authoritative полем.

История позволяет добавлять, изменять и удалять замеры с compare-and-swap.
Прогресс сравним только внутри серии с одинаковыми рукой, оружием и
длительностью. Эти записи пока не входят в remote sync.

### `month_participants`

Связь участника с конкретным `(year, month)`. Удаление этой связи убирает
человека из месяца, но не удаляет `Participant` и историю других месяцев.

### `attendance`

Отметка принадлежит `(year, month, day, participant_id)`. Составной FK
запрещает attendance без membership этого месяца.

### `participant_day_markers`

Семантическая отметка `(year, month, day, participant_id)`: bitmask событий
`Payment`, `SpecialTraining`, `FirstVisit`, `Other`, `LedTraining` и plain-text
note до 500 символов. `LedTraining` означает, что этот участник вёл тренировку
именно в эту дату; это не свойство профиля. Цвет и значок — только
UI-представление. Persisted-маркер `SpecialTraining = 0x02` показывается
пользователю как «Тренировка в доспехах»; его значение не меняется ради
совместимости существующих БД.
Составной FK на membership удаляет отметки вместе с участником месяца. FK на
`month_days` намеренно нет: отметка выключенной даты сохраняется.

### `EventRecord`

Турниры — отдельный bounded context и отдельная локальная SQLite-БД. Для
новой установки обе БД лежат в `QStandardPaths::AppLocalDataLocation`; если
обнаружен старый `test_data.db` в рабочем каталоге, `events_data.db` создаётся
рядом с ним. Неявного перемещения существующего журнала нет. Запись содержит
UUID, название, дату, явную категорию, общую plain-text заметку, список бойцов
клуба, отдельный список сопровождающих и строки боёв. Категория различает
клубный турнир на тренировке, выездное соревнование и СМБ-турнир (мягкий).
Сторона боя — либо UUID внутреннего участника, либо свободное имя внешнего
человека; счёт хранится двумя целыми числами.

Физический FK между двумя SQLite-файлами невозможен. Поэтому БД турниров
хранит UUID внутреннего участника и snapshots отображаемого имени и ФИО. UUID
сохраняет логическую связь, snapshots не дают старому турниру потерять
контекст после переименования или отсутствия основной БД. Aggregate читается в
одной SQLite read transaction; `revision` и compare-and-swap блокируют тихую
перезапись более новой версии другим процессом. Турниры пока не входят в
remote sync. Event schema v2 расширяет snapshot отображаемого имени с 200 до
300 символов. Schema v3 добавляет индекс истории участника для вычисляемой
статистики турниров и боёв; v1 и v2 мигрируют последовательно и транзакционно
с проверкой внешних ключей. Schema v4 добавляет `event_category` и индекс по
категории и дате. Старые записи мигрируют как `Unspecified`: автоматически
считать их выездными или клубными запрещено, поскольку это исказило бы
историю. При следующем сохранении старого события UI требует выбрать
категорию. Schema v5 расширяет диапазон категорий для СМБ и добавляет
`event_attendees`; миграция v4 -> v5 сохраняет события, бойцов и бои, а новый
список оставляет пустым.

`event_participants` содержит только людей, которые соревновались.
`event_attendees` содержит тех, кто ездил с клубом, но не соревновался. Роли
взаимоисключающие внутри одного события. Наличие строки боя не является
обязательным фильтром участия: протокол боёв может быть заполнен не полностью.
Связь через `event_attendees` увеличивает только отдельный счётчик поездок и
не увеличивает число турниров участника.

### `MonthSnapshot`

Содержит:

- явное состояние `Missing`, `Ready` или `Error` и текст ошибки;
- participants текущего месяца: UUID, вычисленное имя и исходные
  `historicalName`/`fullName`;
- active days;
- attendance.
- day markers.

Snapshot применяется целиком только в sync-сценариях. Обычное сохранение UI
делает upsert attendance и не меняет membership/profile.

### Статистика участника

Статистика является вычисляемым read-model, а не отдельным хранилищем. Она
читается по стабильному `ParticipantId` из journal и event bounded contexts и
не синхронизируется как самостоятельное authoritative state.

- месячная история строится по `month_participants`, включая месяцы с нулём
  посещений;
- посещением считается `attendance.is_checked = 1` только на активной дате;
- выключенные даты, сохранённые для возможного восстановления, в статистику
  не входят;
- тренировка в доспехах считается только при одновременной отметке
  `SpecialTraining` и посещении;
- `LedTraining` считается отдельно на активных датах;
- ручной месяц начала тренировок определяет приблизительный календарный стаж,
  но не добавляет вымышленные посещения и не меняет среднее по журналу;
- первая и последняя даты означают только границы записанных посещений;
- турниром считается присутствие UUID в составе `event_participants`, боем —
  присутствие UUID на любой стороне строки боя;
- клубные турниры, выездные соревнования, СМБ-турниры и старые события без
  категории считаются раздельно;
- присутствие в `event_attendees` считается как поездка без участия и не
  влияет на число турниров или боёв.

Замеры ударов остаются отдельной историей исходных измерений. Представление
показывает вычисленную скорость и изменение внутри сопоставимой серии:
одинаковые рука, оружие и длительность. Смешивать разные условия в один
показатель прогресса нельзя.

Процент посещаемости не вычисляется: `month_participants` не хранит точную дату
вступления внутри месяца. Отдельная БД статистики и ручной массовый пересчёт
запрещены, поскольку создадут рассинхронизированный второй source of truth.

## SQLite schema v12

- `participants`: вычисленное `display_name`, optional `historical_name`,
  `full_name`, `birth_day`, `birth_month`, `birth_year`, `rank`, `contact`,
  `combat_hand`, nullable-пару `training_start_year`/
  `training_start_month`, nullable `club_joined_on`, `notes`, `archived_at`;
- `participant_rank_history`: составной ключ `(participant_id, rank)`,
  optional `obtained_on`, каскадное удаление вместе с участником;
- `month_participants`;
- `attendance`;
- `month_days`;
- `participant_day_markers`;
- `participant_emblems`: один lazy PNG BLOB на участника, SHA-256, размеры и
  revision;
- `participant_strike_tests`: история исходных замеров ударов и revision.

Обязательны:

- `PRAGMA foreign_keys = ON` для каждого connection;
- `CHECK` для year/month/day/boolean;
- unique primary keys;
- индексы чтения месяца и истории участника;
- `PRAGMA user_version = 12`.

Schema v2 мигрирует последовательно
v2 -> v3 -> v4 -> v5 -> v7 -> v8 -> v9 -> v10 -> v11 -> v12.
Schema v3-v5 также мигрируют последовательно и транзакционно. Для существующих
миграция v4 -> v5 назначает `rank = 'guest'`; v5 -> v7 назначает пустые ФИО и
контакт и расширяет допустимую marker mask с 15 до 31. Миграция v7 -> v8
сохраняет прежний `display_name` как `historical_name`; v8 -> v9 добавляет
`combat_hand = 'unknown'`; v9 -> v10 добавляет nullable-пару месяца начала
тренировок; v10 -> v11 добавляет герб и замеры ударов; v11 -> v12 добавляет
дату вступления и нормализованную историю званий. Unversioned legacy schema
сначала мигрирует в v3, затем до v12:
`dd.MM.yyyy` сохраняет год, а `dd.MM` привязывается к единственному году
настройки `month_days`.
Неоднозначное соответствие месяца нескольким годам отклоняется. Profile
triggers проверяют согласованность вычисленного, историчного и полного имени,
контакт, notes, звание, календарную корректность birthday и согласованность
nullable-пары начала тренировок.

Промежуточная development-схема с profile-column `is_trainer` также называлась
v6, но имеет другой контракт. При открытии ограничение маркеров обновляется
без потери данных, значения старого флага архивируются, устаревшая колонка
удаляется, затем схема мигрирует через v7, v8, v9, v10, v11 до v12.

`saveActiveDays()` не удаляет attendance и day markers выключенных дней. При
повторном включении сохраненные данные восстанавливаются.

## Month setup contract

Чтение месяца не создаёт данные. `IJournalStorage::getMonthState()` отделяет
отсутствующий месяц от пустого настроенного месяца и от ошибки storage. Для
нового local-месяца UI предлагает создать его с нуля или атомарно перенести
участников и дни из другого месяца. При ошибке чтения редактирование блокируется,
чтобы пустой UI не перезаписал существующие данные.

В сформированный месяц можно атомарно добавить участников из любого другого
сформированного месяца. Merge выполняется по UUID: существующий состав,
attendance и day markers сохраняются, совпадающие участники пропускаются.
Источник выбирается из списка, полученного через `IJournalStorage::listMonths()`.
Результат merge хранится в `MainWindow` как in-memory `MonthSnapshot`; storage
не изменяется до явного `Сохранить месяц`. При навигации или закрытии окна UI
обязан предложить сохранить, отбросить или оставить черновик открытым.

Компактная панель главного календаря также использует список сформированных
месяцев. Текущий несформированный месяц добавляется в selector временно, а
стрелки остаются отдельным способом перейти к произвольному новому месяцу.

## Sync contract

Push/pull работают с `MonthSnapshot`, а не с `name + day`.

- Push заменяет remote membership/active days/attendance/day markers месяца.
- Pull заменяет local membership/active days/attendance/day markers месяца.
- Для неизвестного UUID создаётся global participant с исходными ФИО и
  историчным именем из snapshot; существующая global-карточка не
  перезаписывается.
- Ошибка remote read не трактуется как пустой месяц.
- Remote writes используют Hrana batch: DML идет по `ok`-conditions, rollback —
  по отрицанию успешного commit.
- Pull читает participants/days/attendance/day markers одной read transaction.
- Обычная загрузка local/remote месяца также использует один
  `loadMonthSnapshot()`; частично прочитанный aggregate не возвращается.
- Contact, birthday, rank, notes, archive status, герб и замеры ударов не входят
  в `MonthSnapshot` и пока не синхронизируются.

Push сначала сохраняет UI attendance в local SQLite и только затем читает
snapshot для отправки. Ошибка remote read при pull не изменяет local aggregate.

`JournalRemote::connect()` по умолчанию только проверяет current schema. DDL и
миграции разрешаются явно через `JOURNAL_BOOTSTRAP_REMOTE_SCHEMA=1` для
одноразового localhost bootstrap.

Текущий HTTP-клиент синхронный и использует вложенный `QEventLoop`. Это PoC,
не production transport. До multi-client нужны async/cancellation, auth, TLS,
month revision/CAS и conflict UI. Без revision текущий full replace имеет
last-writer-wins и может потерять параллельные изменения.
