# Импорт старого журнала Excel

Утилиты предназначены для контролируемого разового импорта старых журналов.
Процесс состоит из двух этапов:

1. Python-конвертер читает структуру конкретного Excel и создаёт проверяемый
   JSON-контракт.
2. `journal_legacy_importer` проверяет JSON и записывает данные через
   `SqliteConnect`. Поэтому схема, ограничения и миграции не дублируются в
   Python-коде.

Исходный Excel, промежуточный JSON и полученная БД содержат персональные данные.
Их следует хранить в `old_xlsx/`; эта папка исключена из Git.

## Одномесячный формат v1 (`try1.xlsx`)

```powershell
python Tools/convert_legacy_attendance.py `
  old_xlsx/try1.xlsx old_xlsx/try1-import.json
build/journal_legacy_importer.exe `
  old_xlsx/try1-import.json old_xlsx/try1-import.db
```

Импортёр намеренно отказывается перезаписывать существующую БД. Для повторного
импорта укажите новый путь или предварительно вручную удалите тестовый результат.

`convert_legacy_attendance.py` привязан к расположению колонок `try1.xlsx`.
Другой формат нужно сначала проверить и адаптировать; молчаливо угадывать
колонки нельзя.

## Многомесячный формат v2 (`посещаемость Армэ.xlsx`)

```powershell
python Tools/convert_arme_attendance.py `
  "old_xlsx/посещаемость Армэ.xlsx" `
  old_xlsx/arme-import-v2.json `
  --identity-map old_xlsx/arme-identity-map.json `
  --report old_xlsx/arme-import-report.txt
build/journal_legacy_importer.exe `
  old_xlsx/arme-import-v2.json old_xlsx/arme-review.db
python Tools/verify_arme_import.py `
  old_xlsx/arme-import-v2.json old_xlsx/arme-review.db
```

Формат v2 содержит единый каталог профилей и массив месяцев. Одинаковый UUID
используется во всех месяцах, поэтому история одного человека не дробится на
копии. Явная таблица алиасов `arme-identity-map.json` объединяет подтверждённые
варианты имён; тёзки и неоднозначные записи остаются разными профилями. Этот
файл тоже содержит персональные данные и должен оставаться в `old_xlsx/`.

Перед записью импортёр полностью проверяет профили, UUID, составы, даты,
посещения и отметки. Каждый месяц сохраняется через транзакционный
`replaceMonth()`, профили — через `updateParticipantProfile()`. Результат сначала
пишется во временную БД в каталоге назначения, повторно читается через storage и
только затем атомарно переименовывается в итоговый файл.

`verify_arme_import.py` независимо, в read-only режиме, проверяет schema version,
`integrity_check`, внешние ключи и точное совпадение всех импортируемых строк:
профилей, истории званий, составов, дней, посещений и отметок.

### Правила преобразования Армэ

- знак `+` означает посещение;
- зелёная заливка создаёт отметку оплаты независимо от посещения;
- голубая заливка, а в поздней легенде светло-жёлтая, означает первую тренировку;
- красная — отметку «Другое» об отсутствии оплаты;
- оранжевая — отметку «Другое» для уточнения;
- фиолетовая — железную тренировку;
- повторяющаяся травма сворачивается в одну отметку на первой дате, а все даты
  перечисляются в примечании;
- `+(?)` считается посещением и сохраняет сомнение в отметке «Другое»;
- текстовые примечания Excel переносятся в примечание к отметке;
- сводный цвет в колонке «Оплачено» не содержит даты: он переносится на
  последний учётный день месяца с явным текстом «точная дата не указана»;
- нижний блок «Рыцари» входит в состав соответствующего месяца;
- подтверждённые алиасы с разными непустыми контактами считаются конфликтом и
  останавливают конвертацию с указанием листов и строк;
- текущее звание берётся из последнего листа человека, а встречавшиеся звания
  попадают в историю без выдуманной точной даты получения;
- неизвестные дата вступления и месяц начала тренировок остаются пустыми.

В исходном файле встречается невалидный цвет границы `rgb="none"`.
`convert_arme_attendance.py` исправляет его только во временной копии ZIP/XML;
пользовательский `.xlsx` не изменяется.
