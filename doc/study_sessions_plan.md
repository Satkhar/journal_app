# Study Sessions Plan

Документ для последовательного разбора проекта до уровня "могу объяснить и изменять как автор".

## Общие правила на все сессии

1. Сначала понять текущий поток данных, потом менять код.
2. Один логический шаг = одна проверка (build + ручной сценарий).
3. После сессии записать 5-10 коротких тезисов в заметки.
4. Если нашли проблему, сначала зафиксировать гипотезу, потом правку.

---

## Сессия 1: Карта проекта и поведение приложения

### Цель
Понять архитектуру на уровне блоков и увидеть реальные пользовательские сценарии.

### Чеклист действий
1. Прочитать `doc/code_detailed_walkthrough.md`.
2. Прочитать `doc/arch.md` и `doc/arch.mmd`.
3. Собрать проект.
4. Запустить приложение.
5. Пройти руками сценарии:
   - `Local` connect
   - `Add User`
   - `Save Current Table`
   - `Push to Server`
   - `Pull from Server`
   - `Remote` connect (read-only)
6. Зафиксировать, какие кнопки активны в каждом режиме.

### Вопросы самопроверки
1. Какие есть слои в проекте и кто за что отвечает?
2. Что именно происходит при `Local` и `Remote` подключении?
3. Почему `Remote` режим read-only?
4. Чем `Read Base` отличается от `refreshMonth()`?

### Результат сессии
Вы можете объяснить архитектуру без чтения кода строка-в-строку.

## Сессия 2: Контракты и use-case слой

### Цель
Понять бизнес-логику и контракты между UI и хранилищами.

### Чеклист действий
1. Разобрать `App/Inc/IJournalStorage.hpp`.
2. Разобрать `App/Inc/JournalApp.hpp`.
3. Разобрать `App/Src/JournalApp.cpp`.
4. Для каждого метода `JournalApp` выписать:
   - входы,
   - выход,
   - побочные эффекты,
   - кто вызывает.
5. Нарисовать mini-sequence:
   - `loadMonth`
   - `saveMonth`

### Вопросы самопроверки
1. Зачем нужен интерфейс `IJournalStorage`?
2. Почему `loadMonth()` не должен создавать данные и зачем нужен `MonthState`?
3. Какие данные держит `JournalApp` как контекст?

### Результат сессии
Вы понимаете use-case слой независимо от SQLite/HTTP деталей.

## Сессия 3: Local storage (SQLite)

### Цель
Разобрать локальное хранилище до уровня SQL-запросов и транзакций.

### Чеклист действий
1. Разобрать `App/Inc/SqliteConnect.hpp`.
2. Разобрать `App/Src/SqliteConnect.cpp`.
3. Разобрать `App/Src/JournalLocal.cpp`.
4. Отдельно понять:
   - `monthPattern` и `dayString`
   - транзакцию `saveMonth` (delete + insert + commit)
   - почему connection name уникальный (`QUuid`)
5. Протестировать повторный `Pull` и `Read Base`.

### Вопросы самопроверки
1. Почему фиксированное имя Qt SQL connection опасно?
2. Что произойдет при ошибке insert внутри `saveMonth`?
3. Почему для PoC используется полная перезапись месяца?

### Результат сессии
Вы можете объяснить local storage и диагностировать SQL-проблемы.

## Сессия 4: Remote storage и SyncService

### Цель
Понять HTTP-протокол, обработку ошибок и sync-потоки.

### Чеклист действий
1. Разобрать `App/Inc/JournalRemote.hpp`.
2. Разобрать `App/Src/JournalRemote.cpp`.
3. Разобрать `App/Inc/SyncService.hpp`.
4. Разобрать `App/Src/SyncService.cpp`.
5. Проверить руками:
   - неверный URL сервера
   - timeout
   - `pull` при недоступном сервере
6. Убедиться, что при неуспешном pull локальная БД не затирается.

### Вопросы самопроверки
1. Как формируется запрос к `/v2/pipeline`?
2. Где хранится и как используется `lastError`?
3. Почему sync-логика вынесена из `MainWindow` в `SyncService`?

### Результат сессии
Вы понимаете, как работает удаленное хранилище и почему sync безопасен.

## Сессия 5: UI-оркестрация и самостоятельная правка

### Цель
Понять orchestration в `MainWindow` и закрепить знания собственной правкой.

### Чеклист действий
1. Разобрать `App/Inc/mainwindow.hpp`.
2. Разобрать `App/Src/mainwindow.cpp`.
3. Разобрать `journal_app.ui`.
4. Выписать таблицу кнопок:
   - активность в `LOCAL/REMOTE`
   - источник чтения
   - куда идет запись
5. Сделать 1 небольшую самостоятельную правку UI/логики.
6. Прогнать build и ручной smoke-test.

### Вопросы самопроверки
1. Какие методы `MainWindow` только про UI, а какие про бизнес-операции?
2. Как и когда обновляется mode badge?
3. Почему `Read Base` должен читать локальную БД всегда?

### Результат сессии
Вы можете уверенно менять UI-логику без поломки архитектуры.
